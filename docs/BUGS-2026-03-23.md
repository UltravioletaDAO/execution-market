# Bug Report & Fixes - March 23, 2026

## Executive Summary

Extensive debugging session on Execution Market identified and resolved 8 critical issues across the full stack - from cron monitoring to reputation systems. All fixes committed and ready for deployment.

## Issues Fixed

### 1. Task Expiration Notification Gap
**Status:** ✅ FIXED  
**Severity:** High  
**Impact:** Operations team not notified when tasks expired with no applications

**Root Cause:**
- Cron monitor running in isolated session couldn't use `message` tool
- Missing notification pathway for expired tasks with zero applications

**Fix:**
- Updated SKILL.md with proper cron session messaging patterns
- Updated HEARTBEAT.md to include notification flow
- Modified cron prompt to use `message` tool correctly
- **Commits:** 2744720, bc76f83

### 2. Wrong Wallet Paying Escrow
**Status:** ✅ FIXED  
**Severity:** Critical  
**Impact:** Platform Wallet incorrectly paying escrow instead of Agent Wallet

**Root Cause:**
- `_get_fase2_client()` hardcoded `WALLET_PRIVATE_KEY` environment variable
- This pointed to platform wallet, not agent-specific wallet

**Fix:**
- Implemented multi-wallet support with `_resolve_payer_wallet()`
- Added `AGENT_WALLET_PRIVATE_KEY` environment variable
- Dynamic wallet selection based on context
- **Commit:** ed36874
- **Deployment Required:** Set `AGENT_WALLET_PRIVATE_KEY` in ECS environment

### 3. Dashboard Stale Cache
**Status:** ✅ FIXED  
**Severity:** Medium  
**Impact:** Dashboard showing outdated transaction data across different tasks

**Root Cause:**
- React Query configured with `staleTime: 5 minutes`
- `refetchOnWindowFocus: false` preventing fresh data on context switches
- Cache persisting across different task views

**Fix:**
- Reduced staleTime to 30 seconds
- Enabled `refetchOnWindowFocus: true`
- Improved cache invalidation on task changes
- **Commit:** ed36874

### 4. Submit Evidence Button Persistence
**Status:** ✅ FIXED  
**Severity:** Medium  
**Impact:** Submit button remained visible after worker submitted evidence

**Root Cause:**
- `handleSubmissionComplete` callback didn't update local task status
- UI state not synchronized with backend submission state

**Fix:**
- Update `selectedTask.status` to 'submitted' on completion
- Trigger refetch to ensure data consistency
- **Commit:** e865740

### 5. Reputation Score > 100
**Status:** ✅ FIXED  
**Severity:** High  
**Impact:** Profile displaying impossible reputation scores (120/100)

**Root Cause:**
- Dynamic scoring algorithm could produce values > 100
- No clamping before database write
- CHECK constraint possibly dropped during schema migration

**Fix:**
- Frontend clamping to 0-100 range
- Backend validation and clamping
- Migration 072 to re-add CHECK constraint: `reputation_score BETWEEN 0 AND 100`
- **Commit:** becd5239

### 6. Missing Agent→Worker Rating (WS-2b)
**Status:** ✅ FIXED  
**Severity:** High  
**Impact:** Agents never rating workers after task approval, breaking reputation loop

**Root Cause:**
- `_send_reputation_feedback` only called in SDK settlement path
- Missing from post-approval side effects in direct API flow

**Fix:**
- Added WS-2b side effect in `_execute_post_approval_side_effects`
- Ensures bidirectional reputation feedback in all approval flows
- **Commit:** 5650234

### 7. Worker Auto-Registration Failure (WS-1)
**Status:** 🔍 UNDER INVESTIGATION  
**Severity:** High  
**Impact:** Workers completing tasks without ERC-8004 identity registration

**Evidence:**
- Worker completed 2 tasks successfully
- No ERC-8004 identity found in registry
- Missing from facilitator transaction logs

**Possible Root Causes:**
- `erc8004_auto_register_worker` feature flag disabled
- Facilitator failing silently during registration
- Submission object missing executor wallet data
- Registration transaction failing but not reported

**Investigation Required:**
- Check feature flag status in production
- Review facilitator logs for registration attempts
- Verify submission data includes executor address
- Test registration flow end-to-end

### 8. No Auto-Assignment by Reputation
**Status:** ✅ FIXED  
**Severity:** Medium  
**Impact:** High-reputation workers not automatically assigned, requiring manual operator intervention

**Root Cause:**
- SKILL.md lacked auto-assignment logic based on ERC-8004 reputation scores

**Fix:**
- Updated SKILL.md to v3.9.0 with smart assignment logic:
  - Workers with reputation ≥ 80% → auto-assign
  - Workers with reputation < 80% or unrated → notify operator
  - New workers with imported reputation → use discounted scores for assignment
