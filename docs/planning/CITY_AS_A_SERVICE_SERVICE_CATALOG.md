## 1. Why this doc exists

The current CaaS planning stack defines the thesis, pilot wedge, operator loop, and go-to-market framing.
What it still needs is a packaging document that translates those ideas into sellable offers with clear delivery rules.

This doc defines:
- the first commercial offers
- what each offer includes and excludes
- intake fields needed to operate them
- pricing logic and surcharge rules
- upsells and follow-on motions
- the recommended pilot order for the Miami-first launch

The goal is not a broad catalog.
The goal is a narrow, implementation-useful menu that an operator can actually sell, scope, and dispatch.

## 2. Packaging principles

### 2.1 Sell outcomes, not generic labor
Customers should buy:
- office reality confirmed
- packet attempt proven
- posting visibly verified
- rejection clearly diagnosed
- redirect path clearly identified

They should not feel like they are hiring an errand runner with vague deliverables.

### 2.2 Start with Miami-first density
The first offers should assume:
- one metro area
- repeated municipal offices
- recurring SMB/property/compliance workflows
- manual operator review on all meaningful outcomes

### 2.3 Keep municipal interpretation bounded
CaaS should package:
- observation
- submission attempts
- structured evidence
- routing clarity
- next-step capture

CaaS should not initially package:
- legal advice
- code interpretation guarantees
- guaranteed permit approval
- agency relationships or influence

### 2.4 Every offer should produce structured next steps
The return payload should always answer:
- what happened
- what proves it
- what likely happens next
- what follow-on task should be created, if any

## 5. Core launch offers

## 5.1 Counter Reality Check

### Best for
- confirming the right office, desk, window, or route
- verifying which form/version is actually in use
- checking whether a city website is outdated
- validating same-day office reality before a larger filing attempt

### Deliverables
- one in-person office visit
- one primary structured question
- one backup clarification question if needed
- office presence proof
- normalized answer with source type: observed / heard / documented
- redirect target if the office routes elsewhere
- operator-facing next-step recommendation

### Required intake fields
- jurisdiction
- office name or suspected office
- office address if known
- workflow type (permit, renewal, posting, inspection, etc.)
- exact question to ask
- acceptable answer format
- urgency window
- fallback instruction if staff refuse to answer directly

### Standard exclusions
- more than one office visit
- legal interpretation of the answer
- guaranteed definitive answer if staff are contradictory
- packet submission during the same order unless explicitly bundled

### Pricing logic
- base fee for local office verification
- same-day urgency surcharge
- high-complexity question surcharge if the ask includes multiple branches
- extra-office surcharge if the worker must visit a second office after redirect

### Typical upsells
- convert directly into Packet Submission Attempt
- add Queue Wait + Handoff if timing matters
- add Office Redirect Follow-Through if bounced

### Success condition
Customer receives a structured answer that materially reduces routing ambiguity for the next step.

## 5.2 Packet Submission Attempt

### Best for
- in-person document drop-off
- permit packet filing
- renewal submission
- proving whether a packet was accepted, rejected, or redirected

### Deliverables
- one in-person submission attempt
- structured outcome: accepted / rejected / redirected / blocked
- receipt, stamped copy, or equivalent acceptance evidence when available
- rejection reason capture when rejected
- redirect target when redirected
- next-step recommendation

### Required intake fields
- jurisdiction
- target office
- workflow type
- packet checklist
- named documents included
- acceptance criteria if known
- whether copies/printouts are already prepared
- what to do if redirected
- what to do if rejected for a fixable reason
- deadline or urgency level

### Standard exclusions
- document drafting or repair work
- legal review of packet quality
- payment of municipal fees unless explicitly included
- more than one submission office in a single base order

### Pricing logic
- higher base fee than Counter Reality Check
- document complexity surcharge based on packet size/class
- queue-time surcharge after included threshold
- urgency surcharge for same-day or cutoff-sensitive filing
- retry/resubmission surcharge for same-order repeat attempts

### Typical upsells
- Rejection Diagnosis + Resubmission Prep
- Office Redirect Follow-Through
- Permit Follow-Through Bundle
- print/copy/notarization coordination add-on later if supported operationally

### Success condition
Customer receives proof of attempt plus a structured classification of what happened and what to do next.

## 5.3 Posting Compliance Check

### Best for
- storefront notices
- construction permit postings
- public notice verification
- recurring proof that a required posting remains visible and legible

### Deliverables
- context photo
- close-up legibility photo
- pass/fail checklist
- timestamped proof bundle
- visibility notes
- next-step recommendation if missing, damaged, or illegible

### Required intake fields
- site address
- posting type
- posting location expectation
- visibility/legibility rules
- recurrence cadence if this is a repeat check
- failure threshold for escalation

### Standard exclusions
- creating or printing the notice itself
- legal sufficiency opinions beyond the checklist
- interior inspections unless separately scoped

### Pricing logic
- base per site
- recurrence discount for scheduled re-checks
- after-hours surcharge if access timing is constrained
- multi-posting surcharge if several notices must be verified separately

### Typical upsells
- Multi-Site Posting Verification Bundle
- Exterior Compliance Sweep
- Permit Follow-Through Bundle when the posting is tied to active filings

### Success condition
Customer receives visible proof that the posting is compliant or a clear failure state with next-step guidance.

## 5.4 Exterior Compliance Sweep

### Best for
- fast exterior condition/compliance checks
- signage visibility review
- pre-inspection readiness snapshots
- property ops spot checks

### Deliverables
- checklist-based exterior audit
- required-angle photo set
- pass/fail by checklist item
- notable exceptions or risks
- escalation recommendation for failed items

### Required intake fields
- site address
- checklist to inspect
- required capture angles
- known risk items
- escalation threshold
- whether comparison against a prior sweep is needed

### Standard exclusions
- interior inspection
- code interpretation guarantees
- remediation work
- direct engagement with inspectors unless separately scoped

### Pricing logic
- base per site visit
- checklist-length surcharge for larger audits
- multi-site bundle pricing if same operator run is feasible
- urgency surcharge for same-day pre-inspection requests

### Typical upsells
- Posting Compliance Check
- Multi-Site Posting Verification Bundle
- Queue Wait + Handoff if the sweep feeds an inspection-day workflow

### Success condition
Customer receives a structured exterior status snapshot that can trigger corrective action before a higher-cost failure.

## 5.5 Queue Wait + Handoff

### Best for
- holding a place in line
- making sure a packet handoff happens before cutoff
- appointment-presence requirements
- document pickup after a hearing or office visit

### Deliverables
- arrival proof
- elapsed wait summary
- handoff outcome classification
- blocked reason if unresolved
- final status and next-step recommendation

### Required intake fields
- office/location
- arrival deadline
- handoff condition
- max wait budget
- cutoff time if any
- fallback instruction if office closes or refuses service

### Standard exclusions
- indefinite waiting without a defined budget
- broader office Q&A beyond the scoped handoff need
- submission retries unless separately authorized

### Pricing logic
- base dispatch fee
- included wait window
- per-increment wait surcharge after included threshold
- deadline/cutoff urgency premium

### Typical upsells
- Packet Submission Attempt
- Counter Reality Check
- Office Redirect Follow-Through

### Success condition
Customer preserves a time-sensitive municipal opportunity they would otherwise lose to queue friction.

## 6. Follow-on / chained offers

## 6.1 Rejection Diagnosis + Resubmission Prep

### Best for
- packets rejected for unclear or partially fixable reasons
- customers who need a structured recovery path, not just a failed attempt notice

### Deliverables
- normalized rejection reasons
- document deficiency summary
- office-specific resubmission notes if obtained
- recommended next task type
- optional resubmission-ready checklist draft

### Intake dependency
Requires a prior Packet Submission Attempt or equivalent rejection evidence.

### Exclusions
- legal drafting
- preparing new legal documents
- guaranteed acceptance on resubmission

### Pricing logic
- lower base if done from prior EM evidence
- add-on fee if a second office clarification is required

## 6.2 Office Redirect Follow-Through

### Best for
- cases where the first office sends the customer elsewhere
- situations where the redirect itself is the main uncertainty

### Deliverables
- redirected office visit
- redirect validation
- new office proof
- updated routing guidance
- final structured status

### Intake dependency
Usually triggered from Counter Reality Check or Packet Submission Attempt.

### Pricing logic
- redirect bundle surcharge rather than full re-intake
- second-office premium if location is materially farther

## 6.3 Permit Follow-Through Bundle

### Best for
- operators who want a defined municipal checkpoint reached, not just one attempt
- recurring SMB/compliance customers with repeated city friction

### Includes
- one Counter Reality Check
- one Packet Submission Attempt
- one follow-on redirect or rejection clarification step
- structured workflow summary at each checkpoint

### Exclusions
- unlimited retries
- legal filing strategy
- agency fee advancement unless separately approved

### Pricing logic
- bundled discount relative to separate orders
- capped included steps with add-on pricing after scope is exceeded

### Why it matters
This is the first offer that begins to sell verified municipal progress instead of isolated tasks.

## 6.4 Multi-Site Posting Verification Bundle

### Best for
- property managers
- contractors with multiple active sites
- operators who need repeat proof across several addresses

### Deliverables
- per-site posting proof bundle
- rolled-up exception summary
- site-by-site pass/fail matrix

### Pricing logic
- base bundle fee plus per-site rate
- route-density discount when addresses cluster geographically

## 7. Intake model

The operator should be able to map all first-wave orders into a common intake structure plus offer-specific fields.

### 7.1 Common intake fields
- customer/account name
- contact method
- jurisdiction
- office/site address
- workflow type
- desired outcome
- deadline / urgency
- acceptable proof type
- fallback instruction if blocked
- permission for follow-on task creation

### 7.2 Offer-specific fields

#### Counter Reality Check
- exact question
- answer format
- whether a redirect target is acceptable as success

#### Packet Submission Attempt
- packet document list
- expected acceptance artifact
- rejection handling rule
- redirect handling rule

#### Posting Compliance Check
- posting type
- required location
- visibility criteria
- recurrence cadence

#### Exterior Compliance Sweep
- checklist items
- must-capture angles
- fail thresholds

#### Queue Wait + Handoff
- arrival deadline
- wait budget
- cutoff time
- exact handoff condition

## 8. Pricing framework

This section is for commercial logic, not final public price points.

### 8.1 Base pricing units
Each order should price from a small set of components:
- dispatch base
- office/site complexity
- wait time
- urgency
- second location / redirect
- document complexity
- recurrence or volume discount

### 8.2 Suggested internal pricing bands

#### Low-complexity
Use for:
- single-site posting checks
- one-question counter verification with low ambiguity

#### Medium-complexity
Use for:
- packet drop-off attempts
- exterior sweeps with moderate checklist length
- short queue waits tied to handoff

#### High-complexity
Use for:
- cut-off-sensitive filings
- redirect-prone offices
- repeat visits within a compressed window
- bundled follow-through offers

### 8.3 Recommended pricing posture
The first pilot should optimize for:
- margin clarity
- speed of quoting
- easy operator judgment

It should not optimize for:
- elaborate custom quotes
- low-price race dynamics
- pretending municipal uncertainty can be priced away entirely

## 9. Upsell and expansion logic

### 9.1 Immediate upsells
- Counter Reality Check -> Packet Submission Attempt
- Packet Submission Attempt -> Rejection Diagnosis + Resubmission Prep
- Redirected submission -> Office Redirect Follow-Through
- Posting Compliance Check -> scheduled recurring verification
- Exterior Compliance Sweep -> Multi-Site bundle

### 9.2 Account expansion motion
A pilot customer should ideally move from:
1. one urgent municipal pain point
2. repeat office/site usage
3. bundled follow-through orders
4. workflow-level municipal support

### 9.3 Signals that upsell is justified
- repeated work at the same office or site
- customer repeatedly asking "what should we do next?"
- customer already coordinating multiple vendors manually
- frequent rejections or redirects that create follow-on value

## 10. Exclusions and guardrails

### 10.1 Initial exclusions
Do not package these in the first pilot catalog:
- legal advice
- guaranteed approvals or permits issued
- inspector influence / relationship claims
- broad “handle everything with the city” retainers
- cash handling or municipal fee advancement by default
- unlimited waiting or unlimited retries

### 10.2 Evidence and claim guardrails
Commercial copy and operator language should avoid claiming:
- certainty when staff answers conflict
- legal sufficiency beyond the checklist/evidence scope
- that city rules are fully standardized across offices

### 10.3 Scope control rule
If an order requires more than:
- one office,
- one primary municipal step,
- or one clear proof objective,

then it should usually be converted into a chained/bundled offer instead of being forced into a single base task.

## 11. Miami-first pilot order

### Phase 1 — launch first
1. Counter Reality Check
2. Packet Submission Attempt
3. Posting Compliance Check

Why:
- closest to the current wedge
- strongest repeatability
- strongest structured-outcome value
- easiest to explain to SMB and permit-ops buyers

### Phase 2 — add once Phase 1 is stable
4. Exterior Compliance Sweep
5. Queue Wait + Handoff

Why:
- expands adjacent demand without diluting the core permit/posting wedge
- still produces structured municipal outcomes

### Phase 3 — bundle only after repeated workflows appear
6. Rejection Diagnosis + Resubmission Prep
7. Office Redirect Follow-Through
8. Permit Follow-Through Bundle
9. Multi-Site Posting Verification Bundle

Why:
- these offers depend on repeat workflow logic and clean follow-on motion
- they become more valuable once office memory and operator review loops mature

## 12. What sales and ops should say clearly

### 12.1 Positioning sentence
**We handle the physical city steps your staff or software can’t complete: office checks, filing attempts, postings, and structured next steps.**

### 12.2 Promise sentence
**We promise a verified attempt window, evidence of what happened, and a structured next-step output — not city cooperation we cannot control.**

### 12.3 Why this matters commercially
This framing:
- preserves trust
- keeps scope bounded
- supports premium pricing relative to generic errands
- aligns with the operator-review and municipal-memory product truth

## 15. Summary

This document adds the missing commercial packaging layer for City as a Service.
It turns the wedge into a sellable menu with:
- concrete offers
- clear deliverables
- required intake fields
- pricing logic
- upsell paths
- exclusions
- a recommended Miami-first rollout order

The key discipline is staying narrow.
