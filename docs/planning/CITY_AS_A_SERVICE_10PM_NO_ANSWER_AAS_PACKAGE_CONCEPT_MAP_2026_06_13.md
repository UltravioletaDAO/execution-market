## Concept-map implementation rules

Use these rules for any future AAS concept expansion while no operator answer exists:

1. **Name one uncertainty, not a service promise.** The package begins as a bounded uncertainty statement, never as customer copy.
2. **Keep evidence language adjacent to uncertainty.** Any method, count, state, or checklist must carry limitations and not-checked fields.
3. **Separate package grammar from authorization.** A concept map may describe modules, but it cannot authorize collection, delivery, publication, dispatch, runtime mutation, reputation, payment, or authority.
4. **Prefer redacted references over raw evidence.** Planning language should assume digest/reference-only evidence until a separate redaction/delivery gate exists.
5. **Carry blocked claims forward.** Every safe claim must remain next to the claims that are still not safe.
6. **Do not cross-promote stopped projects.** Dream work stays inside Execution Market AAS / City-as-a-Service only.
