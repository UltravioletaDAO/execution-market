### Facilitator Nonce Issue

The first run failed with nonce errors when operations were sent < 5s apart:

```
ErrorResp { code: -32000, message: "nonce too low: next nonce 4571, tx nonce 4569" }
ContractCall("TxWatcher(Timeout)")
```

**Workaround:** 15-second delay between escrow operations.

**Root cause:** The Facilitator's nonce manager doesn't wait for TX confirmation
before reusing nonce slots. Same issue existed in V1 settle flow (was fixed there
with a configured delay). V2 escrow scheme needs the same fix.

**Orphaned escrow:** $0.05 USDC locked in TX `0x69acb1c4...` from the failed
first run. Refund failed due to nonce error. Needs manual resolution or will
remain locked until escrow expiry.

## SDK Bugs Found and Fixed

### Bug 1: Double `0x` in `_compute_nonce()` (CRITICAL)

`Web3.keccak().hex()` returns `"0xabc..."` (with prefix) in web3 >= 6.x.
SDK did `"0x" + Web3.keccak(...).hex()` → `"0x0xabc..."`. Breaks all signing.

**Fix:** `raw.hex().removeprefix("0x")` before prepending `"0x"`.

### Bug 2: `bytes32` encoding in `_sign_erc3009()` (CRITICAL)

`encode_typed_data()` in eth_account >= 0.10.0 expects `bytes` for bytes32
fields. SDK passed hex string → treated as ASCII bytes (66+ chars) → exceeds
32-byte limit → `ValueOutOfBounds` crash.

**Fix:** `bytes.fromhex(nonce.removeprefix("0x"))` before passing to encoder.

### Bug 3: Double `0x` in signature return (MINOR)

Same HexBytes issue as Bug 1. `signed.signature.hex()` may include `0x`.

**Fix:** Conditional prefix: `sig if sig.startswith("0x") else "0x" + sig`.

### Resolution

- All 3 fixes applied to `uvd-x402-sdk-python` (v0.13.1)
- Published to PyPI: `pip install uvd-x402-sdk==0.13.1`
- `execution-market` dependency bumped to `>=0.13.1`
- Full bug report: [`SDK_FEEDBACK_FOR_X402R.md`](./SDK_FEEDBACK_FOR_X402R.md)

## Conversation with x402r Team

### Ali Abdoli's Question (2026-02-11 22:54 UTC)

> "I'm just confused what this form of escrowing accomplishes, since it seems
> like the agent puts funds in the escrow and it itself can release or take
> the funds out, meaning it has full control over the escrow so all the funds
> anyways"

### Our PaymentOperator Configuration

```
PaymentOperator (0xb963...):
  authorizeCondition = UsdcTvlLimit     ← anyone can lock funds (protocol safety only)
  releaseCondition   = StaticAddressCondition(Facilitator)  ← ONLY Facilitator
  refundCondition    = StaticAddressCondition(Facilitator)  ← ONLY Facilitator
  feeCalculator      = address(0)       ← we charge 8% ourselves
  escrowPeriod       = 0                ← no mandatory lock period
  freezePeriod       = 0                ← no freeze
```

### What the Current Escrow DOES Protect

1. **Fund locking** — agent's USDC is locked on-chain, can't be spent elsewhere
   while task is active
2. **Designated receiver** — release goes ONLY to the address set at authorize
   time (contract-enforced, not even Facilitator can change destination)
3. **Designated refundee** — refund goes ONLY to original payer
   (contract-enforced)
4. **Transparency** — all operations visible on BaseScan, labeled "x402 Transaction"
5. **Gasless** — Facilitator pays all gas, neither agent nor worker needs ETH

### What the Current Escrow DOES NOT Protect (Ali's Point)

**Ali is correct.** The current model is NOT fully trustless:

- **The Facilitator decides** when to release vs refund
- **Execution Market controls the Facilitator** (we run it)
- Therefore, **the platform has full control** over locked funds
- A malicious platform could release to itself or refuse to refund

The trust model is: **workers trust the platform**, not the smart contract.
The escrow provides transparency and fund-locking, but not trustless arbitration.

### Why This Matters

We've been describing x402r escrow as "trustless" in our documentation and
articles. This is **inaccurate** for our current configuration. The correct
description is:

- **Transparent, gasless, on-chain escrow** with fund locking ✅
- **Platform-controlled release/refund** (trust in platform) ⚠️
- **NOT trustless** in the sense that neither party can be cheated ❌

## Trustless Escrow Gap Analysis

### Current State (Fase 2)

```
Trust Model:
  Agent   → trusts Platform to release when work is good
  Worker  → trusts Platform to release when work is submitted
  Platform → has full control via Facilitator
```

### What Fully Trustless Requires

For the escrow to be truly trustless, **no single party should have unilateral
control** over fund release or refund. This requires:

1. **Multi-party release conditions** — both agent AND worker (or a neutral
   arbiter) must agree on release
2. **Automatic timeout refund** — if no action is taken within X days, funds
   auto-refund to agent (no Facilitator needed)
3. **On-chain dispute resolution** — if agent and worker disagree, a neutral
   mechanism resolves it
4. **Worker-triggered release** — worker can prove completion and trigger
   release without platform involvement

### Options for Trustless Escrow

#### Option A: Multi-sig Conditions

Replace `StaticAddressCondition(Facilitator)` with a condition that requires
signatures from BOTH the agent and a neutral verifier (or the worker + agent):

```
releaseCondition = MultiSigCondition([agent, facilitator])
refundCondition  = MultiSigCondition([agent, facilitator]) OR TimeoutCondition(30 days)
```

**Pro:** True 2-of-2 multi-sig. **Con:** More complex UX, requires agent to sign release.

#### Option B: Optimistic Release with Dispute Window

```
releaseCondition = OptimisticCondition({
  proposer: facilitator,
  disputeWindow: 48 hours,
  disputeResolver: arbitration_contract,
})
```

Worker submits proof → Platform proposes release → 48h window for agent to
dispute → if no dispute, auto-release. If disputed, goes to on-chain arbitration.

**Pro:** Good UX (auto-release if no dispute). **Con:** 48h delay on every payment.

#### Option C: Escrow Period + Automatic Release

Use the `escrowPeriod` parameter (currently 0) to set a mandatory lock:

```
escrowPeriod = 7 days
// After 7 days, anyone can trigger release (or auto-refund if unclaimed)
```

**Pro:** Uses existing x402r contract feature. **Con:** Fixed delay, no early release.

#### Option D: Hybrid (Recommended for Fase 3)

Combine approaches:
- **Normal path:** Platform releases immediately (same as today, fast UX)
- **Dispute path:** If worker or agent disputes, enters arbitration
- **Timeout path:** If platform goes offline, auto-refund after 30 days
- **Evidence hash:** feedbackHash on-chain proves what evidence was submitted

This gives fast UX in the normal case while protecting against platform failure
or malicious behavior.

## Trustless Escrow Roadmap (Fase 3)

> **CRITICAL: This is NOT a "nice to have" — this is a fundamental promise
> of the platform that must be delivered.**

### Phase 3.1: Timeout Safety Net (Quick Win)

**Goal:** Protect against platform failure (Facilitator goes offline).

- Set `escrowPeriod > 0` in PaymentOperator (e.g., 30 days)
- After escrow period expires, anyone can trigger refund
- No contract changes needed — just config change in PaymentOperator

**Effort:** 1-2 days (deploy new PaymentOperator with non-zero escrowPeriod)

### Phase 3.2: On-Chain Evidence Anchoring

**Goal:** Make feedback documents verifiable on-chain.

- Store actual `feedbackHash` (keccak256 of evidence JSON) on-chain
  (currently `0x0000...0000`)
- Dashboard `/feedback/:taskId` resolves to the evidence document
- Anyone can verify: hash(document) == on-chain feedbackHash

**Effort:** 3-5 days (feedback_store already computes hash, just need to pass it)

### Phase 3.3: Dispute Resolution Contract

**Goal:** Neither party has unilateral control over disputed funds.

- New `DisputeCondition` contract for releaseCondition
- Normal flow: Platform proposes release, 24h dispute window
- Dispute flow: Locked until arbiter (DAO vote, oracle, or multi-sig) decides
- Timeout: Auto-refund after 30 days if unresolved

**Effort:** 2-4 weeks (new Solidity contract + integration)

### Phase 3.4: Fully Trustless Release

**Goal:** Worker can trigger release with proof, no platform needed.

- `ProofOfCompletionCondition` — worker submits evidence hash that matches
  the task requirements hash
- Agent pre-defines acceptance criteria on-chain at task creation
- If criteria match, auto-release (no human/platform in the loop)
- This is the "end game" — fully autonomous task execution

**Effort:** 1-2 months (significant contract + UX work)

### Priority Matrix

| Phase | Impact | Effort | Priority |
|-------|--------|--------|----------|
| 3.1 Timeout Safety | High (failure protection) | Low (config change) | **P0 — Do immediately** |
| 3.2 Evidence Anchoring | Medium (verifiability) | Low (code exists) | **P1 — This sprint** |
| 3.3 Dispute Resolution | High (trustless) | High (new contract) | **P2 — Next sprint** |
| 3.4 Proof of Completion | Very High (fully autonomous) | Very High | **P3 — Roadmap** |

## feedbackUri Resolution Gap

The `feedbackUri` values recorded on-chain in this test session don't resolve.

**Why:** The E2E test sends feedback directly to the Facilitator, bypassing the
MCP server's `persist_and_hash_feedback()` which creates the actual document.

**In production:** The MCP server creates the feedback JSON → uploads to S3 →
stores in Supabase → passes the CDN/API URL as `feedbackUri` to the Facilitator.
The dashboard route `/feedback/:taskId` then resolves correctly.

**Fix applied:** Test script updated to use API URL format
(`api.execution.market/api/v1/reputation/feedback/{id}`).

**Remaining work:** Future E2E tests should go through the MCP server's
`/workers/rate` and `/agents/rate` endpoints to test the full document creation
flow.
